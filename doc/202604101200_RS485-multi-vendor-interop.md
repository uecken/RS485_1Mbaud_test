# 異なるモジュール / メーカーの RS-485 を相互接続する際の注意点

作成日: 2026-04-10
位置付け: 本プロジェクト (RS-485 1Mbaud 無線化) で異なるベンダー・異なる電源電圧の RS-485 トランシーバモジュールを混在させる場合の注意事項と落とし穴をまとめた技術リファレンス。

## 結論 (TL;DR)

| 観点 | 結論 |
|------|------|
| 規格 (TIA/EIA-485-A) 適合品同士は通信可能か? | **原則 Yes**。差動電圧 ±1.5V 以上 / 受信しきい値 ±200mV / コモンモード -7V〜+12V を満たせば相互接続可能 |
| 実務上の落とし穴 | **A/B ラベルの極性、終端抵抗、バイアス抵抗、Fail-Safe、DE/RE 制御方式、絶縁/非絶縁、電源電圧 (3.3V vs 5V)** で問題が起きる |
| 本プロジェクトへの影響 | **送受信で異なるモジュールを混在させる場合、必ず事前に組合せ動作確認が必要**。電気的には繋がっても、DE/RE 制御の方式違いで動かない例が実際に発生している |

## 1. RS-485 規格 (TIA/EIA-485-A) の電圧仕様

| 項目 | 規格値 | 説明 |
|------|--------|------|
| 送信側 差動出力電圧 V_OD (54Ω 負荷) | **±1.5V 〜 ±5V** | A-B 間の電圧差 |
| 送信側 コモンモード電圧 V_OC | -1V 〜 +3V | A/B の中点電圧 |
| 受信側 入力しきい値 V_IT | **±200mV** | 受信側はこの差で論理判定 |
| 受信側 入力電圧範囲 V_CM | **-7V 〜 +12V** | コモンモード許容範囲 |
| 論理表現 | V_A > V_B → 論理 1 (MARK / アイドル) / V_A < V_B → 論理 0 (SPACE) | 規格 + UART 慣例 |

→ **規格適合品同士は論理的に互換性がある**。ただし、規格内で許容される範囲があるため、実装の差で問題が起きうる。

## 2. 異なるモジュール・メーカーで起きる問題と注意点

### 2.1 A / B ラベルの極性が逆 [最も多い落とし穴]

**問題**: メーカーやデータシートによって A と B のラベルが逆の場合がある。

| メーカー / 系統 | A の意味 |
|---------------|--------|
| **TI (SN65HVD 系)** | A = Non-inverting (+) ← 規格通り |
| **Maxim/Analog Devices (一部)** | A = Inverting (-) ← **規格と逆** の場合あり |
| **MaxLinear (SP3485 等)** | A = Non-inverting (+) ← 規格通り |
| **その他汎用モジュール** | データシート未記載のものもあり |

**判定方法**:
1. データシート の Functional Diagram で「A は DI=HIGH 時に H 出力か L 出力か」を確認
2. データシートが不明な場合は **A-A / B-B 接続で動かなければ A-B / B-A に入れ替えてみる** (実機動作確認が確実)
3. 送信側で `0xFF` 連続送信し、A-GND 間電圧をオシロで観測。**送信中に A 電圧 > B 電圧** ならラベル正しい

**対策**:
- **送受信で同一型番のモジュールを使う** (最も確実)
- 異なるモジュールを使う場合は、事前にデータシートで A/B 極性を確認
- ターミナル基板に **A/B 入替ジャンパ** を設けておくと現地調整が楽

---

### 2.2 電源電圧 (3.3V vs 5V) の差動出力電圧マージン

| IC | 電源 | 差動出力 V_OD (54Ω 負荷) |
|----|-----|----------------------|
| MAX485, SP485EE, LTC1485 | **5V** | **≥2.0V** |
| MAX3485, SP3485, SN65HVD75 | **3.3V** | **≥1.5V** |

**問題**:
- 3.3V IC は規格ギリギリ (1.5V) → **長距離・ノイズ環境ではマージン不足**
- 5V IC は余裕あり → **長距離・ノイズ環境に強い**
- 5V IC 送信 → 3.3V IC 受信は問題なし (受信しきい値は ±200mV)
- 3.3V IC 送信 → 長距離 (例 100m以上) → **ケーブル損失でしきい値割れの可能性**

**対策**:
- **長距離・ノイズ環境では送信側に 5V IC** (LTC1485 等) を選ぶ
- 3.3V IC を送信側に使う場合は **120Ω 終端抵抗を必ず両端に追加** (反射防止 + 負荷整合)
- ESP32 と直結する場合は 3.3V IC が便利 (レベル変換不要)

---

### 2.3 終端抵抗 (120Ω) の有無

**問題**: モジュールによって 120Ω 終端抵抗が **基板上に実装済み or 未実装** で異なる。

| パターン | 結果 |
|---------|------|
| 両端ともに **未実装** | 反射発生 → 高速通信で波形なまり、誤受信 |
| 両端ともに **実装済み** | 正しい (規格通り) |
| 両端のうち **片方だけ実装済み** | 一方の終端のみで反射が部分的に残る |
| **複数モジュールが実装済み** | 過剰負荷で送信側が駆動しきれない (V_OD 低下) |

**対策**:
- **バス両端のみに 120Ω 終端を 1 つずつ** が原則 (合計 2 個)
- モジュールに終端が実装済みかを **テスターで計測** (DE OFF 状態で A-B 間抵抗を測る)
- 中間のモジュールには **絶対に追加終端しない**
- バス上に 3 台以上接続するスタブの場合は終端方式を再検討

---

### 2.4 バイアス抵抗 (Fail-Safe バイアス) の有無

**問題**: アイドル時 (送信側全員が DE=OFF の状態) に、A/B 線が浮き状態になり差動電圧が 0V → 受信側がノイズで誤判定するリスク。

**Fail-Safe バイアス**:
- A 線を Vcc 側にプルアップ、B 線を GND 側にプルダウン (例: 680Ω)
- アイドル時に V_A > V_B となり、論理 1 (MARK / アイドル) を維持

**モジュール別の対応**:

| IC / モジュール | Fail-Safe | 備考 |
|---------------|----------|------|
| **SN65HVD75 (TI)** | **内蔵** ✅ | 外付け不要 |
| SP3485 (MaxLinear) | 内蔵 ✅ | 外付け不要 |
| MAX3485 (Analog Devices) | **なし** ❌ | **外付けバイアス必要** |
| MAX485 (Analog Devices) | なし ❌ | 外付けバイアス必要 |
| SP485EE (MaxLinear) | なし ❌ | 外付けバイアス必要 |
| 汎用モジュール (AliExpress 等) | **モジュール次第** | 基板上に R 実装の有無を確認 |

**対策**:
- **Fail-Safe 内蔵 IC を選ぶ** (SN65HVD75 / SP3485)
- 内蔵なしの IC を使う場合は、バス上のどこか 1 箇所に 680Ω バイアス抵抗を追加
- バスの **両端のターミネータと組み合わせて、バイアス + 終端 + バイアス** の構成が一般的

---

### 2.5 DE / RE 制御方式の違い [本プロジェクトの実害]

**問題**: 半二重 RS-485 では送受切替の DE/RE ピン制御方式が **モジュールによって大きく異なる**。

| モジュール | DE/RE 制御方式 | 高速通信 |
|-----------|--------------|--------|
| **MAX3485 / SP3485 / SN65HVD75 系の素 IC モジュール** | **手動 GPIO 制御** | **○** (1Mbaud 以上対応) |
| **M5Stack RS485 Unit (SP485EE 搭載)** | **TX データ連動の自動切替 (Q1: SS8050、SP485EE 内蔵スルーレート制限)** | **△** TX 側で 500 kbaud 上限。**RX 側のみなら 1 Mbaud OK** |
| **Atomic RS485 Base** | 同上 (M5Stack RS485 Unit と同回路) | 同上 |
| **M5StickC RS485 HAT** | 自動切替 | **×** (115,200bps 仕様上限) |
| **M5Stack 絶縁型 RS485 ユニット (ISO485 / CA-IS3082W)** | **自動切替 + 絶縁、CA-IS3082W はスルーレート制限なし** | **○** TX/RX とも 1 Mbaud OK |

**本プロジェクトでの最終確認結果** ([202604092200_RS485-baudrate-eval-results.md](202604092200_RS485-baudrate-eval-results.md), [202604092230_RS485Unit-vs-ISO485Unit-comparison.md](202604092230_RS485Unit-vs-ISO485Unit-comparison.md)):

- **TX = ISO485 Unit (CA-IS3082W) + RX = M5Stack RS485 Unit (SP485EE)** の混在構成で **1 Mbaud 連続送信 FER 0%** を確認 (uart_tx/uart_rx の rolling counter pattern, Arduino + ESP-IDF UART API 直叩き、burst 32 frame/loop)
- 真のボトルネックは IC でも DE/RE 回路でもなく **Arduino HardwareSerial.read() の 1-byte 律速** だった。ESP-IDF `uart_read_bytes()` のバッチ読みに切り替えた時点で 1 Mbaud 受信が普通に通った
- SP485EE 側 (M5Stack RS485 Unit) を **送信** に使うと、SP485EE 内蔵のスルーレート制限と DE/RE 自動切替の応答遅延で 500 kbaud が実用上限。**受信専用なら 1 Mbaud OK**
- ISO485 Unit (CA-IS3082W) は スルーレート制限がないため、TX/RX どちらでも 1 Mbaud で安定動作

**対策**:
- 高速通信 (≥1 Mbaud) では **TX 側に CA-IS3082W (ISO485) または手動 DE/RE 制御モジュール** を採用する
- 受信側は SP485EE 系 (M5Stack RS485 Unit) でも 1 Mbaud は通る
- ファームウェアは **HardwareSerial.read() ではなく ESP-IDF UART API 直叩き** (`uart_read_bytes` バッチ) を使うこと
- 自動切替モジュールを使う場合は **送信用途で速度の上限を実機テスト** で必ず確認する

---

### 2.6 絶縁 / 非絶縁の混在

**問題**: 絶縁型 (ISO485 等) と非絶縁型を混在させると、GND 電位差・グラウンドループの扱いが変わる。

| パターン | 結果 |
|---------|------|
| 絶縁 × 絶縁 | GND 共通化不要、グラウンドループなし |
| 非絶縁 × 非絶縁 | **GND 共通化が必要** (XLR 5pin の Pin 1 = GND を結線) |
| **絶縁 × 非絶縁** | 絶縁側の意味が薄れる (非絶縁側で GND ループが発生する可能性) |

**対策**:
- 全モジュールを **絶縁型** に統一する (ノイズ環境で安全)
- 非絶縁型を使う場合は **必ず GND を共通化** する (XLR Pin 1 を結線)
- 絶縁と非絶縁を混在させる場合は、絶縁側の意味を理解した上で構成する

---

### 2.7 受信負荷 (Unit Load) と最大接続台数

**問題**: モジュールによって受信入力の負荷が異なり、バス上に接続できる台数の合計に制限がある。

| IC | Unit Load | 最大接続台数 |
|----|-----------|----------|
| MAX485 / MAX3485 / SP3485 / SP485EE / SN65HVD75 / LTC1485 | **1/8** または **1/4** | **32台** または **128台** |
| 古い IC | 1 (フル UL) | 32 台 |

**対策**:
- 本プロジェクトは 2 台 (送信側・受信側) のみのため問題なし
- 多台接続する場合は IC のデータシートで Unit Load を確認

---

### 2.8 スルーレート (立上り時間) の混在

**問題**: スルーレート制限あり IC とスルーレート制限なし IC を混在させると、**遅い側が律速** する。

| IC | スルーレート制限 | 立上り時間 | 最大ボーレート (実用) |
|----|--------------|----------|-------------------|
| MAX485, SP485EE | あり | ~30ns | 500-750 kbaud (短距離) |
| MAX13487E | あり | ~800ns | **500 kbaud 上限** |
| MAX3485, SP3485, SN65HVD75 | **なし** | **~4-7ns** | **10 Mbps** |

**対策**:
- 高速通信では **送受信ともにスルーレート制限なし IC** に統一
- 低 EMI が必要な環境ではスルーレート制限ありを使うが、速度を犠牲にする

## 3. 異なるモジュール混在時のチェックリスト

新しいモジュール組合せを採用する前に、以下を必ず確認する:

| # | チェック項目 | 確認方法 |
|---|------------|---------|
| 1 | A/B ラベルの極性が一致するか | データシート Functional Diagram、または DI=HIGH 時の A/B 出力電圧を実測 |
| 2 | 電源電圧 (3.3V or 5V) | データシート、PCB シルク、テスターで実測 |
| 3 | 終端抵抗が基板に実装されているか | DE OFF 時の A-B 間抵抗を実測 (~120Ω なら実装済) |
| 4 | バイアス抵抗が基板に実装されているか | 無信号時の V_A, V_B を Vcc 基準で実測 |
| 5 | Fail-Safe の有無 | データシート、または無信号時の RX 出力を観測 |
| 6 | DE/RE 制御方式 (手動 GPIO / 自動切替) | データシート、PCB 回路図、Q1 の有無 |
| 7 | 絶縁 / 非絶縁 | データシート、PCB 上の絶縁部品 (フォトカプラ・トランス) 有無 |
| 8 | スルーレート制限の有無 | データシート (Driver Slew Rate Limiting / Rise Time) |
| 9 | 受信 Unit Load (バスへの負荷) | データシート (1/8 UL or 1/4 UL) |
| 10 | 最大ボーレート (IC仕様 vs モジュール仕様) | データシート、製品ページ |

## 4. 推奨構成 (本プロジェクト)

### 安全策: 同一型番で統一

| 用途 | 推奨モジュール | 数量 |
|------|-------------|------|
| 送信側 | **Soldered SN65HVD75 Breakout** ×1 | 1 |
| 受信側 | **Soldered SN65HVD75 Breakout** ×1 | 1 |

理由:
- 全員 SN65HVD75 (3.3V, 20Mbps, Fail-Safe 内蔵, 手動 DE/RE)
- A/B 極性、Fail-Safe、Unit Load で悩む必要なし
- ESP32 と直結可能 (3.3V)
- OSHW で回路図公開、トラブル時に検証可能

### 妥協案: ベンダー混在

| 用途 | 推奨モジュール |
|------|-------------|
| 送信側 | Waveshare SP3485 Board (3.3V) |
| 受信側 | Soldered SN65HVD75 Breakout (3.3V) |

→ **両方とも 3.3V / 手動 DE/RE / Fail-Safe 系**であれば実用上問題なし。
ただし **必ず事前に組合せ動作確認** を行うこと。

### 不採用: 自動方向制御モジュール

| モジュール | 理由 |
|-----------|------|
| M5Stack RS485 Unit (SP485EE) | DE/RE 自動切替で実用 250 kbaud 上限 |
| M5StickC RS485 HAT | 仕様上限 115,200 bps |
| M5Stack 絶縁型 RS485 ユニット (ISO485) | 連続送信で破綻 (条件依存) |

## 5. 本プロジェクトで実際に起きた失敗例

### 5.1 SP485EE 搭載モジュールで 750kbaud NG → 真因は HardwareSerial RX 律速

詳細は [課題と進捗.md §0](課題と進捗.md#0-rs-485-物理層のボーレート上限) と [202604092200_RS485-baudrate-eval-results.md](202604092200_RS485-baudrate-eval-results.md) を参照。

| 観察事実 | 当初の推測 | **実際の原因 (最終確認結果)** |
|---------|---------|---------|
| 750 kbaud で受信不可 | MAX485 のスルーレート制限 (公称 MAX485 と思っていた) | **筐体を開けたら SP485EE (10 Mbps 対応) だった**。次の仮説は DE/RE 自動方向制御回路だったが、最終的な真因は別 (下記) |
| 連続送信で FER 多発 (初期測定) | DE/RE 自動切替の追従不良 + データパターン依存 | **真の律速は Arduino HardwareSerial.read() の 1-byte ループ**。ESP-IDF `uart_read_bytes()` のバッチ読みに切り替えた時点で 1 Mbaud 受信が普通に通った |
| **ISO485 (TX) + M5Stack RS485 Unit (RX) の混在で 1 Mbaud 連続送信** | ハードウェア混在で破綻するのでは | **FER 0% で通った** (uart_tx/uart_rx の rolling counter pattern、Arduino + ESP-IDF UART API 直叩き、burst 32 frame/loop) |
| SP485EE 搭載モジュールを **送信側** に使うと 500 kbaud が上限 | DE/RE 回路と IC 内蔵スルーレート制限の複合要因 | 同上。**受信専用なら 1 Mbaud OK**。ISO485 (CA-IS3082W) はスルーレート制限なしのため TX/RX とも 1 Mbaud OK |

**教訓**:
- **モジュールの IC 刻印を必ず実物で確認** する (公称チップ ≠ 実装チップの場合あり)
- **送信側 = ISO485 (CA-IS3082W) / 受信側 = SP485EE 系 OK** の混在構成が現実的
- **HardwareSerial.read() ではなく ESP-IDF `uart_read_bytes()` のバッチ読み** をファームウェアで使うこと。これがないと SP485EE の限界を測る前に Arduino のソフト律速で破綻して見える
- 「ハードウェアの限界」と決めつける前に、**真のボトルネックがソフト側にないか必ず確認** する
