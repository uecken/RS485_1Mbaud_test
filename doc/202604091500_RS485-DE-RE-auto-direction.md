# M5Stack RS485 Unit の自動方向制御回路の問題

## 概要

M5Stack RS485 Unit は、DE/RE (Direction Enable / Receive Enable) ピンをTXデータに連動させて自動的に送信/受信モードを切り替える回路を搭載している。この回路が高ボーレート (>500 kbaud) で通信エラーの原因となる。

**自動方向制御を採用する理由:**
- RS-485 半二重通信では、送信しない時にDE=LOWにしてバスをハイインピーダンスにしないと、他ノードの信号と衝突する
- 通常はDE/RE制御に専用GPIOが1本必要だが、M5StickCのGroveコネクタにはTX/RXの2ピンしかない
- TXデータに連動させることで、**GPIOを追加せずにGroveの2ピンだけで半二重RS-485通信を実現**している
- デイジーチェーン (Dynamixel等) や低速用途では便利

**制約:**
- RS-485 は衝突検出・再送機能を持たず、通信の信頼性はマスタ/スレーブ方式の上位プロトコルに依存する (詳細は後述の比較表を参照)

## 補足: RS-485 の衝突制御と他方式との比較

DE/RE 自動方向制御回路は CSMA/CD ではない。バス監視・衝突検出・再送のいずれも行わず、TXデータのHIGH/LOWに機械的にDE/REを連動させているだけである。衝突回避は上位プロトコル (マスタ/スレーブ方式) が担当する。

| | RS-485 (M5Stack RS485 Unit) | Ethernet (CSMA/CD) | EtherCAT |
|---|---|---|---|
| **規格** | TIA-485 (物理層のみ) | IEEE 802.3 | IEC 61158 (Ethernet ベース) |
| **トポロジ** | マルチドロップバス | 共有バス (10BASE-T以前) / スター (現在) | **リング / デイジーチェーン** |
| **通信方式** | 半二重 | 半二重 (共有バス) / 全二重 (スイッチ) | **全二重** |
| **キャリアセンス** | なし | **あり** (送信前にバス監視) | 不要 (マスタ制御) |
| **衝突検出** | なし | **あり** (送信中に信号比較) | **なし** (衝突が構造的に発生しない) |
| **衝突時の動作** | データ破壊、上位層に任せる | ジャム信号送出 → ランダム待機 → 再送 | 衝突なし |
| **再送制御** | なし (上位プロトコルが担当) | **あり** (最大16回、指数バックオフ) | 不要 |
| **送信権の制御** | 上位プロトコル (マスタ/スレーブ) | 各ノードが自律的に判断 | **マスタが全制御** (フレーム巡回方式) |
| **方向制御** | DE/REピンで明示的に切替 | 不要 (PHYが自動処理) | 不要 (全二重) |
| **同時送信の結果** | **データ破壊** (検出不可) | **検出→再送** | **発生しない** |
| **速度** | ~10 Mbaud | 10M/100M/1G/10Gbps | **100 Mbps** (Ethernet物理層) |
| **リアルタイム性** | 上位プロトコル依存 | 非決定的 (衝突でジッタ) | **決定的** (µsオーダー) |

> - **RS-485**: 物理層のみの規格で衝突保護なし。上位プロトコル (Dynamixel, Modbus RTU 等) のマスタ/スレーブ方式で衝突を**防止**する設計。
> - **Ethernet (CSMA/CD)**: 衝突を**検出して回復**する設計。現在はスイッチで全二重化されCSMA/CDは使われない。
> - **EtherCAT**: マスタがフレームをリング上に巡回させ、各スレーブが通過中に読み書きする方式。衝突が**構造的に発生しない**設計。ロボット制御で広く使用。

## RS-485 半二重通信の方向制御

RS-485 半二重通信では、トランシーバの DE/RE ピンで送信/受信を切り替える:

```
DE (Driver Enable):  HIGH=送信有効  LOW=送信無効(ハイインピーダンス)
RE (Receiver Enable): LOW=受信有効  HIGH=受信無効

通常は DE と RE を結線し、1本の信号で制御:
  DE=RE=HIGH → 送信モード (A/Bに信号出力、受信無効)
  DE=RE=LOW  → 受信モード (A/Bはハイインピーダンス、受信有効)
```

## M5Stack RS485 Unit の自動方向制御回路

### 回路図 (SP485EEN周辺)

```
                    +5V
                     │
                    R3 (4.7KΩ)
                     │
TX ─── R4(1KΩ) ─── Q1 base      U1 (SP485EEN)
                    (SS8050 NPN)   ┌──────────┐
                     │             │ 1  RO    │── RX (受信データ出力)
                  Q1 collector ──│ 2  RE    │
                     │           ──│ 3  DE    │
                    GND            │ 4  DI    │── TX (送信データ入力)
                                   │ 5  GND   │
                                   │ 6  A     │── RS485_A
                                   │ 7  B     │── RS485_B
                                   │ 8  VCC   │── +5V
                                   └──────────┘
```

### 動作原理

```
TX = LOW (スタートビット, データ'0'):
  → Q1 base = LOW → Q1 OFF
  → collector = HIGH (R3でプルアップ)
  → DE=HIGH, RE=HIGH → 送信モード ✓

TX = HIGH (アイドル, データ'1'):
  → Q1 base = HIGH → Q1 ON
  → collector = LOW (GNDに引く)
  → DE=LOW, RE=LOW → 受信モード ✗ (送信中なのに受信に切替!)
```

### 問題: データの'1'ビットごとにDE/REが切り替わる

UART NRZ で 0xAA (10101010) を送信する場合 (LSB first):

```
         Start  b0  b1  b2  b3  b4  b5  b6  b7  Stop
TX:       L     L   H   L   H   L   H   L   H   H
Q1:       OFF   OFF ON  OFF ON  OFF ON  OFF ON   ON
DE/RE:    TX    TX  RX! TX  RX! TX  RX! TX  RX!  RX
          ↑送信  ↑送信 ↑受信! ↑送信 ↑受信!
```

**ビットごとにトランシーバが送信/受信モードを切り替える。**

### なぜ低ボーレートでは動作するのか

DE/RE切替にはRC時定数がある (R3=4.7KΩ × Q1の寄生容量 + SP485EENの入力容量):

```
低ボーレート (例: 115200 baud):
  ビット幅 = 8.68µs
  DE/REの応答時間 ~数百ns
  → ビット幅 >> DE/RE応答時間
  → DE/REが完全には切り替わらず、中間電位で送信モードを維持
  → 動作する

高ボーレート (例: 1 Mbaud):
  ビット幅 = 1.0µs
  DE/REの応答時間 ~数百ns
  → ビット幅に対してDE/RE応答時間が大きい
  → DE/REが受信モードに切り替わり、送信信号が途切れる
  → 通信エラー
```

### データパターンによる影響の違い

```
0x00 (00000000): Start=L, data=全L, Stop=H
  TX: L L L L L L L L L H
  DE: TX TX TX TX TX TX TX TX TX RX
  → ストップビットのみRX。データ部分はすべてTXモード → 比較的安全

0xFF (11111111): Start=L, data=全H, Stop=H
  TX: L H H H H H H H H H
  DE: TX RX RX RX RX RX RX RX RX RX
  → データ部分がほぼすべてRXモード → 信号が出力されない

0xAA (10101010): 
  → 最悪パターン。ビットごとにTX/RX切替
```

## テスト結果との対応

| テスト | 結果 | 説明 |
|--------|------|------|
| Grove直結 (トランシーバなし) 1 Mbaud | **OK** | DE/RE回路が存在しない |
| SP485EEN経由 500 kbaud (0xAA パケット) | **OK** | ビット幅2µsでDE/REが追従しきれず送信維持 |
| SP485EEN経由 600 kbaud (0xAA パケット) | **NG** | ビット幅1.67µsでDE/RE切替が影響開始 |
| SP485EEN経由 1 Mbaud (0x00 パターン) | **要確認** | データ部がすべて0ならDE常時TX |

## 各種 RS-485 モジュールの DE/RE 制御方式

| モジュール | DE/RE制御 | 方式 | 高速(>500kbaud)対応 |
|-----------|----------|------|-------------------|
| **M5Stack RS485 Unit** | 自動 (TX連動) | Q1トランジスタ回路 | NG |
| **M5Stack Atomic RS485 Base** | 自動 (TX連動) | 同様の回路と推定 | NG (推定) |
| **M5Stack RS485 HAT** | 自動 (TX連動) | 同様の回路と推定 | NG (推定) |
| **Waveshare RS485 Board** | **GPIO制御** | DE/REピンが外に出ている | **対応可** |
| **SparkFun RS485 Breakout** | **GPIO制御** | DE/REピンが外に出ている | **対応可** |
| **自作 (MAX3485等)** | **GPIO制御** | 自分で配線 | **対応可** |

> M5Stack系のRS-485モジュールはGroveの2ピン制約により、すべてDE/RE自動切替を採用している。
> 高速通信 (>500 kbaud) にはDE/REがGPIOに出ている汎用ブレイクアウトボードか自作が必要。

## 対策

### 1. DE/RE を固定する (ハードウェア改造)

送信側のRS485 UnitでDE/REをVCC(5V)に直結し、常時送信モードにする。
受信側のRS485 UnitでDE/REをGNDに直結し、常時受信モードにする。

```
送信専用に改造:
  Q1のcollector-R3接続を切断
  DE/RE (pin 2,3) を VCC に直結
  → 常時送信モード (受信不可)

受信専用に改造:
  Q1のcollector-R3接続を切断
  DE/RE (pin 2,3) を GND に直結
  → 常時受信モード (送信不可)
```

**利点**: 最も確実。部品変更不要、配線変更のみ。
**欠点**: 半二重通信不可。片方向専用になる。

### 2. DE/RE を ESP32 の GPIO で制御する

DE/RE をQ1から切り離し、ESP32の空きGPIOで直接制御する。
送信前にDEをHIGH、送信完了後にLOWにする。

```
ESP32 GPIO26 (例) ──→ DE/RE (pin 2,3)
                       (Q1のcollectorから切り離す)

コード:
  digitalWrite(DE_PIN, HIGH);  // 送信モード
  RS485Serial.write(data, len);
  RS485Serial.flush();         // 送信完了待ち
  digitalWrite(DE_PIN, LOW);   // 受信モード
```

**利点**: 半二重通信を維持。タイミング制御が正確。
**欠点**: ハードウェア改造が必要。GPIO1本追加。

### 3. SP3485 搭載モジュール (Atomic RS485 Base) に変更

Atomic RS485 Base は DE/RE の制御方式が異なる可能性がある (要回路図確認)。
ただし SP3485EE もスルーレート制限ありのため、DE/RE問題が同様に存在する可能性あり。

### 4. スルーレート制限なしトランシーバ + GPIO制御で自作

MAX3485 や SN65HVD75 を使い、DE/RE は ESP32 GPIO で制御する自作モジュール。

**利点**: 最高性能。1 Mbaud 以上対応。
**欠点**: 自作が必要。

## 検証結果 (2026-04-09)

### 0x00 vs 0xFF パターンテスト @ 1 Mbaud, RS-485 5m ツイストペアシールド

| パターン | DE/REの状態 | 受信バイト数 | 内訳 | 結果 |
|---------|-----------|------------|------|------|
| **0x00** (全ビット0) | ほぼ常時送信モード | **180 B/s** | x00=0, other=全数 | データ化けあるが届く |
| **0xFF** (全ビット1) | ほぼ常時受信モード | **0 B/s** | - | 何も届かない |

### 結論

**DE/RE自動切替回路が高ボーレート通信失敗の主因と確定。**

- 0x00 (TX=LOW=送信モード維持) → バイトは届く (ただし化けあり → スルーレート制限の複合影響)
- 0xFF (TX=HIGH=受信モード維持) → 一切届かない → **トランシーバが送信していない**
- Grove直結 (DE/RE回路なし) → 1 Mbaud 正常 → ESP32 UART は問題なし

### 残課題

- 0x00 パターンでもデータ化け (x00=0, other=全数) → スタート/ストップビットの H→L / L→H 遷移時にDE/REが一瞬切り替わっている可能性
- オシロスコープで DE/RE ピン波形を確認すればさらに詳細が判明する
